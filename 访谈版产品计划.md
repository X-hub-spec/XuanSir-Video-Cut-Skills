# 访谈版产品计划

## 1. 产品定位

访谈版是一个独立入口的多人多机位粗剪工作台。

它不替代现有口播粗剪入口，而是作为一个新的实验入口存在。等访谈版稳定后，再考虑和原入口合并。

一句话定位：

```text
把多机位访谈素材变成一篇可阅读、可划线、可重排的采访稿，再导出多轨剪辑工程。
```

访谈版服务的核心场景：

- 多人访谈
- 播客录制
- 圆桌对谈
- 课程对谈
- 主持人与嘉宾问答
- 多机位知识类内容

它的核心价值不是自动导播，而是让用户不用面对复杂时间线，也能快速完成内容粗剪。

## 2. 独立入口

访谈版需要独立入口，避免影响现有口播流程。

建议新增入口名称：

```text
访谈粗剪
```

也可以作为别名支持：

```text
多机位访谈
访谈版粗剪
处理这个访谈
帮我剪这个多机位访谈
```

建议目录结构：

```text
访谈粗剪/
├── SKILL.md
├── scripts/
│   ├── prepare_interview_project.js
│   ├── generate_interview_review.js
│   ├── interview_review_server.js
│   ├── import_synced_xml.js
│   ├── detect_timecode.js
│   └── interview_utils.js
└── README.md
```

短期内，访谈版和口播版入口独立。

长期稳定后，再合并成统一入口：

```text
选择项目类型：
1. 单人口播
2. 多人访谈
3. 字幕精修
```

## 3. 访谈项目创建方式

访谈版第一步不是直接转录，而是先创建访谈项目。

访谈项目可以通过三种方式创建。

## 3.1 从已对轨 XML 创建

这是专业用户的优先入口。

用户已经在 Final Cut Pro、DaVinci Resolve 或 Premiere 里完成对轨，然后把 XML 给系统。

系统处理流程：

```text
读取 XML
→ 找到素材文件
→ 解析素材在时间线上的位置
→ 反推出每条素材 offset
→ 生成 interview_project.json
```

优势：

- 最稳定
- 尊重用户已有工作
- 不需要重新猜同步关系
- 适合已有专业后期流程的用户

同步信息记录为：

```json
{
  "syncMethod": "xml",
  "syncConfidence": 1
}
```

## 3.2 从素材 Timecode 创建

适合现场录制规范较好的访谈。

系统读取每个素材的内嵌 timecode：

```text
cam_a: 10:00:00:00
cam_b: 10:00:03:12
mic_a: 10:00:01:08
```

然后换算每条素材相对全局时间轴的 offset。

同步信息记录为：

```json
{
  "syncMethod": "timecode",
  "syncConfidence": 0.95
}
```

如果部分素材没有 timecode：

- 有 timecode 的素材自动对齐。
- 没有 timecode 的素材进入手动校准。
- 页面明确提示“该素材缺少 timecode”。

## 3.3 从普通素材手动创建

适合普通创作者。

用户提供多个视频和音频：

```text
机位A
机位B
张三麦
李四麦
```

系统先创建项目，所有素材 offset 默认为 0。

用户在审核页里手动对齐。

同步信息记录为：

```json
{
  "syncMethod": "manual",
  "syncConfidence": 0.8
}
```

## 4. 对轨策略

访谈版的对轨优先级：

```text
1. 已对轨 XML
2. 素材内嵌 timecode
3. 手动 offset
4. 波形自动建议
5. 多锚点 / 漂移校正
```

MVP 先完成前三个。

波形自动建议和漂移校正可以放到后续版本。

每条素材都要记录同步来源：

```json
{
  "id": "cam_a",
  "offset": 1.24,
  "syncMethod": "xml",
  "syncConfidence": 1
}
```

可选的 `syncMethod`：

```text
xml
timecode
manual
waveform
anchor
unknown
```

这样用户能清楚知道：

- 哪些轨是可靠同步的
- 哪些轨需要检查
- 哪些轨是手动校准过的

## 5. 数据结构

访谈版使用独立项目文件：

```text
interview_project.json
```

不要直接混用口播版的 `review_project_state.json`。

建议项目结构：

```json
{
  "mode": "interview",
  "projectTitle": "访谈粗剪",
  "sources": [
    {
      "id": "cam_a",
      "kind": "video",
      "path": "/abs/cam-a.mp4",
      "name": "正面机位",
      "offset": 0,
      "syncMethod": "xml",
      "syncConfidence": 1,
      "primary": true
    },
    {
      "id": "mic_zhang",
      "kind": "audio",
      "path": "/abs/zhang.wav",
      "name": "张三麦",
      "speakerId": "zhang",
      "offset": 0,
      "syncMethod": "xml",
      "syncConfidence": 1
    }
  ],
  "speakers": [
    {
      "id": "zhang",
      "name": "张三",
      "color": "#2f6fe4"
    }
  ],
  "words": []
}
```

审核状态单独保存：

```text
interview_review_state.json
```

状态文件包含：

- 删除 token
- 文稿修改
- 说话人名称
- 素材 offset
- 当前预览机位
- 当前播放时间
- 段落顺序
- 字幕分段状态

## 6. 审核页结构

访谈审核页继承现有审稿体验，但保持独立界面。

页面仍然是左右结构：

```text
左侧：视频监看 + 多机位控制
右侧：访谈文稿 + 字幕视图
```

## 6.1 左侧：访谈监看区

左侧包含：

- 主视频预览
- 当前预览机位切换
- 素材同步状态
- 每条素材 offset
- 说话人列表
- 当前删除统计
- 当前播放时间

同步状态需要明确显示：

```text
正面机位    XML 对轨       100%
侧面机位    XML 对轨       100%
张三麦      Timecode       95%
李四麦      手动校准       80%
```

切换预览机位时，必须保持当前全局时间点不变。

例如用户正在审核张三第 2 分 10 秒的一句话：

```text
正面机位 → 侧面机位
```

页面仍然停在同一句话，不让用户重新找位置。

## 6.2 右侧：访谈文稿区

文稿显示方式：

```text
张三：我觉得这个事情其实...
李四：对，但我的看法是...
张三：所以我们最后要...
```

每个人使用稳定颜色。

颜色只做辅助，不要让正文变成花哨的时间线。

支持能力：

- 单击文字跳转
- 双击删除 / 取消删除
- Shift 画线批量删除
- 当前播放词高亮
- AI 建议删除
- 文稿修改
- 词库追加
- 撤销 / 重做
- 字幕视图分段 / 合并

## 7. 话轮和段落

访谈版需要区分两个概念。

## 7.1 话轮 Turn

同一个说话人连续说的一段话。

例子：

```text
张三连续说 20 秒 = 一个 turn
```

话轮适合做：

- 说话人标注
- 小范围删除
- 快速定位某个人说的话

## 7.2 段落 Paragraph

由多个话轮组成的一个主题片段。

例子：

```text
张三解释问题
李四回应
张三总结
= 一个 paragraph
```

段落适合做：

- 上移
- 下移
- 调整叙事顺序
- 导出时整体重排

MVP 规则：

```text
按说话人切话轮
按长停顿切段落
支持段落上移 / 下移
```

后续可以支持：

- 合并话轮为段落
- 拆分段落
- 拖动段落排序
- 移动到某段之前 / 之后

## 8. 删除和重排逻辑

删除仍然按 token 工作。

```text
用户画掉文字 token
→ token 变成删除区间
→ 播放预览跳过
→ 导出时移除这些时间段
```

访谈版中，同一个删除区间会同时作用于所有视频轨和音频轨。

段落移动采用全局重排语义：

```text
移动一段话 = 移动这段话对应的全局时间区间
```

也就是说：

- 这段话里的所有机位一起移动。
- 这段话里的所有音频轨一起移动。
- 不允许只移动某个人声音，而画面不动。
- 不允许只移动字幕，而声音不动。

这样可以避免音画错位。

## 9. AI 粗剪建议

访谈版 AI 建议和口播版不同。

口播版主要识别：

- 口误
- 重复
- 静音
- 语气词
- 残句

访谈版还可以识别：

- 无效寒暄
- 长时间跑题
- 主持人无意义回应
- 两人抢话导致不可用片段
- 问题重复
- 回答绕圈
- 可作为精华片段的高价值回答

MVP 先保守：

```text
自动标长静音、重复、口误、残句
不自动删除观点内容
```

观点内容只做提示，不默认选中。

## 10. 导出能力

访谈版 MVP 导出：

- 多轨 FCPXML
- SRT
- 删除区间 JSON
- interview_project.json
- interview_review_state.json

FCPXML 需要表达：

- 主机位主轨
- 其他机位 video lane
- 人声麦克风 audio lane
- 删除后的连续时间线
- 段落重排后的新顺序

访谈版 MVP 不直接渲染 mp4。

原因：

- 访谈用户通常还要在 FCP 里选机位。
- 自动导播规则复杂，容易不符合创作者审美。
- 多轨工程比直接成片更适合专业后期。

后续可以新增“自动导播成片”，但它应该是独立能力。

## 11. 和口播版的关系

短期保持双入口：

```text
口播粗剪入口
访谈粗剪入口
```

共享底层能力：

- token 时间戳
- 删除区间计算
- SRT 导出
- FCPXML 时间映射
- 状态保存
- 文稿 UI 组件

不共享：

- 用户入口
- 项目状态文件
- 多机位准备流程
- 访谈专用 AI 规则

长期等访谈版稳定后，再合并入口。

## 12. MVP 范围

第一版访谈粗剪 MVP 做这些：

- 独立入口：`访谈粗剪`
- 独立项目文件：`interview_project.json`
- 独立状态文件：`interview_review_state.json`
- 支持从 `interview_sources.json` 创建项目
- 支持从已对轨 FCPXML / XML 导入 offset
- 支持读取素材 timecode 生成 offset
- 支持手动 offset 校准
- 支持说话人命名
- 支持访谈文稿显示说话人
- 支持画线删除
- 支持段落上移 / 下移
- 支持多轨 FCPXML + SRT 导出
- 不支持直接渲染 mp4
- 不做自动声纹识别
- 不做自动导播

## 13. 后续版本路线

## v1：稳定访谈工程

- 独立入口
- 项目创建
- XML 对轨
- Timecode 对轨
- 手动 offset
- 说话人文稿
- 画线删除
- 段落移动
- 多轨 FCPXML + SRT

## v2：降低配置成本

- 可视化导入素材
- 自动读取视频/音频元信息
- offset 页面校准
- 多机位缩略预览
- 项目创建向导

## v3：半自动剪辑智能

- 自动识别当前说话人
- 根据说话人推荐主机位
- 根据停顿和抢话提示冲突
- 输出建议导播轨
- 用户可修改建议导播结果

## v4：创作者个人习惯

- 记住某类口误处理习惯
- 记住说话人常用名称
- 记住 BGM 和字幕风格
- 针对某个创作者形成固定粗剪策略

## 14. 最终体验

用户导入一个访谈项目，系统先帮他对轨、转写、标出说话人。

用户进入审核页后，不再面对复杂多轨时间线，而是看到一篇清楚的访谈稿。

他像编辑文章一样：

- 划掉废话
- 删掉停顿
- 调整段落顺序
- 保留高价值回答

最后导出一个多轨 FCPXML，进入 Final Cut Pro 做机位选择、包装和精剪。

访谈版和口播版的区别：

```text
口播版：帮一个人快速删废话，必要时直接出粗剪片。
访谈版：帮多人多机位快速整理内容结构，导出专业多轨工程。
```
